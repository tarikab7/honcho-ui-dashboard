## 2024-10-24 - Hardcoded Internal IP Leak and Custom Markdown Parser Stored XSS
**Vulnerability:** A hardcoded internal IP address (`192.168.178.61`) was found in the fallback logic of `getHonchoApiUrl`. Additionally, the custom `parseMarkdown` function was vulnerable to Stored XSS because it applied regex replacements to generate HTML without first escaping the input.
**Learning:** Hardcoding IPs in static files not only violates security policies by exposing internal network topology but also makes the application inflexible. Furthermore, implementing custom markdown parsers using regex is inherently risky. If HTML escaping is not explicitly performed *before* the markdown transformations, malicious XSS payloads embedded in the markdown will be executed when the HTML is rendered.
**Prevention:** Always use environment variables or dynamic location detection (e.g., `window.location.hostname`) for API endpoints instead of hardcoded IPs. When rendering user-supplied or AI-generated content to HTML, use established, secure markdown libraries if possible. If a custom parser is required, strictly enforce HTML sanitization (e.g., using `escapeHtml`) on the raw input before any structural HTML transformations occur.
## 2026-06-29 - Inconsistent Sanitization and Brittle Custom Escaper Stored XSS
**Vulnerability:** Multiple Stored XSS vulnerabilities were identified in `index.html` where API data (such as `p.id` in `loadPeers` and `senderLabel` in `renderChatHistory`) was directly interpolated into `.innerHTML` assignments without prior HTML entity encoding. Additionally, the custom `escapeHtml` sanitization function was brittle—it used `String.prototype.replace` without checking or casting non-string types, causing runtime errors or silent bypasses when processing nullish or non-string inputs.
**Learning:** Even internal API identifiers (like IDs and labels) should be treated as untrusted data when reflecting them into the DOM. Relying on custom `replace`-based HTML escapers is risky unless input types are rigorously handled and cast to strings. When escaping mechanisms crash on unexpected types, it can prevent pages from rendering securely or fully.
**Prevention:** Always explicitly wrap all variables interpolated into `.innerHTML` with a robust sanitization function (e.g., `escapeHtml(value)`). When implementing custom HTML escaping, ensure the input is safely cast to a string (e.g., `String(unsafe)`) and explicitly handle nullish values to prevent `TypeError` crashes during execution.
